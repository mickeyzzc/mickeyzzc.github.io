From Mermaid to WebGPU: A Blogger's Guide to Frontend Graphics

July 7, 2026 AI Tools Frontend Visualization, SVG, Canvas, WebGL, Mermaid, Data Visualization, Blog Tools 2864 words 14 min read

Why I Went Down This Rabbit Hole

About a year ago, I started building this blog seriously. Over that year, I created dozens of diagrams—architecture diagrams, sequence diagrams, data comparison charts, cost bar charts, technology evolution flowcharts. Then I hit an embarrassing realization:

I’ve been using Mermaid all along, but I had no idea what technology powers it.

So I spent a few weeks surveying the entire frontend graphics stack. From HTML table hacks to WebGPU parallel rendering, from Flash plugins to Mermaid’s diagram-as-code paradigm.

This isn’t an encyclopedia entry. I won’t list every API or give you install commands for every library. This article answers exactly one question: as a blogger (or any developer who needs to render graphics on the frontend), when you face a concrete requirement, what should you pick, why that one, and where are its boundaries?

The article is organized in three tiers—beginner, intermediate, advanced. You can read it top to bottom, or jump straight to the tier you need. Each tier ends with an explicit “Is this enough? Should you keep going?” checkpoint. Skip what you don’t need.

Beginner Tier: Mermaid + Chart.js Solve 90% of Your Needs

Let me start with the conclusion: for blogging, or any small-to-medium frontend charting needs, the Mermaid + Chart.js combo covers 90% of scenarios. The remaining 10% is when you need to go deeper.

Structural Diagrams: Mermaid Is “Markdown for Diagrams”

The most common diagram type in blogs is structural—architecture diagrams, sequence diagrams, flowcharts, ER diagrams. These diagrams share one trait: “the relationships are clear; no visual gymnastics needed.”

Mermaid’s design philosophy can be summed up in one sentence: describe diagrams with Markdown-style text, and let the engine auto-layout and output SVG.

I picked Mermaid as this blog’s structural diagram solution for simple reasons:

  1. Written in Markdown, no external image files—never worry about “where did the image go?”
  2. Version-controllable—git diff shows what changed in your diagram
  3. Natively supported by GitHub/GitLab/Notion/Obsidian—write it, see it
  4. AI-generation friendly—asking an LLM to output a ```mermaid code block is a hundred times easier than asking it to draw
  5. SVG output—vector scaling is lossless, retina displays stay sharp, search engines can index it

An unexpected discovery: Mermaid gets its name from The Little Mermaid, which creator Knut Sveidqvist’s kid was watching at the time. He created Mermaid after losing a Visio file and thinking, “what if diagrams were just text?” This marks the beginning of the “docs-as-code” movement—a design philosophy more inspiring than the technology itself.

When is Mermaid not enough?

  • Diagrams with hundreds of nodes—the layout engine (dagre) starts struggling, and line crossings become unavoidable
  • When you need pixel-level layout control—Mermaid can’t do the manual adjustments that Draw.io offers
  • When you need strict C4 model hierarchy constraints—Structurizr or D2 are better fits
  • When you need hand-drawn/whiteboard-style collaborative sketches—Excalidraw

If your needs don’t cross these boundaries, stick with Mermaid. Don’t switch.

Data Charts: Chart.js, The 68KB Lightweight King

The other common diagram type in blogs is data-driven—comparison bar charts, trend line charts, proportion pie charts. These need: “feed in data, get out a decent-looking chart, and keep it light.”

I recently wrote an article comparing costs of domestic LLMs. The data was API pricing and GPU configs for a few models, and I needed a simple bar chart. Here’s how I evaluated:

NeedAnswer
Chart typeBar chart (only one type)
Data volume6 data points
InteractionHover tooltip showing values
Size constraintKeep it light—blog load time matters
Learning costGet it done in 5 minutes

Check, check, check: Chart.js. Gzipped at only 68KB, the API takes five parameters:

  type: 'bar',
  data: { labels: ['GLM-5','Kimi K2.5','MiniMax M2.7'], datasets: [{ data: [0.5, 1.0, 0.8] }] },
  options: { plugins: { tooltip: { enabled: true } } }
});

When is Chart.js not enough?

  • You need chart types beyond the basic 8 (candlestick, map, Sankey, radar, etc.) → switch to ECharts
  • Data exceeds tens of thousands of points → ECharts has progressive rendering and data sampling
  • You need complex interaction (zoom, brush selection, drill-down) → ECharts has these built in
  • Your site is already heavy, chart weight doesn’t matter anymore → ECharts has the most chart types and the best animations

Core trick: Chart.js shines because it’s light and fast. Don’t mistake “light” for “weak.” For blog scenarios, “just enough” is optimal.

At this tier: Mermaid for structural diagrams, Chart.js for data charts, CSS animations for simple effects (covered later). This combo starts at zero cost and is enough forever. Reading further means you’re actively choosing higher, more complex needs.

Intermediate Tier: When Defaults Aren’t Enough

This tier tackles “not so simple” demands: data grows, interactions get complex, Mermaid’s auto-layout starts irritating you.

ECharts: The “Full Suite” of Chart Types

The essential difference between ECharts and Chart.js isn’t “which is stronger”—it’s positioning:

  • Chart.js is an SUV—great for commuting, light and efficient
  • ECharts is a truck—hauls anything, but heavy itself (360KB)

When do you trade the SUV for the truck?

Case: dashboard visualization. Imagine you’re no longer comparing a few numbers in a blog post—you’re building an ops dashboard. You need line charts overlaid with scatter plots, map heatmaps, real-time data refresh, zoom and brush interaction. Chart.js can’t do this.

ECharts’ core strengths:

  • Most chart types: 50+ varieties, from basic bars to Sankey diagrams and 3D globes
  • Big data optimization: progressive rendering + data sampling, handles millions of data points
  • Dual renderers: Canvas (default, high performance) and SVG (optional, lossless scaling)
  • Fine-grained animations: every element, every property can have independently configured transitions

The cost is size and complexity. ECharts’ configuration options are so extensive they’re practically their own system. A simple chart can balloon into surprisingly long code. If you don’t genuinely need its capabilities, Chart.js is less hassle.

D3.js: Not “Stronger Chart.js”—It’s a “Toolbox of Chart Parts”

The most common misunderstanding about D3 is: “Chart.js isn’t powerful enough, so I should learn D3.” Wrong.

D3 is not a chart library. It’s a data-driven low-level graphics toolkit. Its core capabilities are exactly three:

  • data join: bind data to DOM elements
  • selections: query and manipulate those elements
  • transitions: animate between element states
javascript

// D3's core operation pattern
svg.selectAll("rect")
   .data(dataset)
   .enter()
   .append("rect")
   .attr("height", d => d * 5);

What can D3 do? Any visual effect you can imagine. The cost: making a bar chart takes 10x longer than Chart.js. The New York Times data visualization team are deep D3 users—they need “visual expressions that no library has predefined.”

Key insight: ECharts is a config-driven chart factory (setOption(config)—declarative rendering). D3 is a data-driven low-level toolkit (build from scratch). You don’t pick D3 because it’s “more powerful”—you pick it because what you’re making has no existing chart type that can express it. It’s like choosing C not because C is stronger than Python, but because you need OS-level control.

Beginner advice: If you can clearly name the chart type you need (bar, line, pie), D3 is not your answer. Chart.js or ECharts are. D3 is your weapon when your need exceeds “what existing chart types can express.”

SVG vs Canvas: Performance Traps and Real-World Truths

When you start using ECharts’ Canvas renderer for a real-time dashboard, or bind hundreds of D3 elements to SVG and notice the page stuttering—that’s when you need to understand the fundamental difference between SVG and Canvas.

Don’t memorize that “SVG is retained mode, Canvas is immediate mode.” Understand through scenarios:

Scenario A: Medium interactive chart (< 2,000 elements, each needs clickability)

Use SVG. Every bar is a DOM node, bind a click event and you can navigate. Search engines read it. Screen readers announce it.

Scenario B: Real-time big data dashboard (> 5,000 data points, per-frame refresh)

Use Canvas. It’s a bitmap, no DOM tree overhead. The tradeoff—what gets drawn is just pixels, you can’t bind a click event to a specific bar. You need to implement hit testing yourself: given mouse coordinates (x, y), search your array to figure out which bar that is.

Scenario C: Massive particles/scatter (100K+ data points)

Canvas can’t keep up either. You need WebGL—covered next.

A counterintuitive fact: most people assume “Canvas is always faster than SVG.” But SVG is actually faster for single-element partial updates. One benchmark showed: with 20,000 nodes, changing just one element, SVG takes only 1/7 the time of Canvas. Because SVG only redraws that one element; Canvas must clear and redraw everything. So the choice isn’t simply “more elements → Canvas”—it also depends on your update pattern: full per-frame redraws, or occasional single-element changes.

Practical rules of thumb:

  • Chart with lots of interaction → SVG
  • Real-time animation frames → Canvas
  • Medium dashboard (thousands of points, relatively frequent updates) → Canvas, with requestAnimationFrame + dirty rectangle updates
  • Relationship graph (hundreds of nodes with connecting edges) → SVG (D3 force-directed graphs are SVG)

At this tier: you now understand the ECharts vs Chart.js division of labor, D3’s true role, and the practical boundaries between SVG and Canvas. For blogging, this tier’s knowledge is enough for all diagram selection decisions. Going further is for when you start doing 3D data visualization or massive-scale rendering.

Advanced Tier: WebGL/WebGPU and the Performance Frontier

This tier handles scenarios where data volume exceeds Canvas’ limits, or you need genuine 3D spatial expression.

When Is Canvas Not Enough?

Canvas 2D’s performance ceiling is roughly 20K-50K elements. Beyond that, the CPU cost of per-frame clear-and-redraw becomes the bottleneck—there’s too much to draw.

WebGL hands rendering to the GPU’s programmable shader pipeline; a single draw call can process hundreds of thousands of vertices. Scenarios:

  • Particle systems (flow field/wind field visualization, ambient background effects) — hundreds of thousands of particles
  • Large-scale point clouds (LiDAR data, geospatial information)
  • 3D scatter plots (need Z-axis depth, rotatable camera)
  • Volume rendering (3D visualization of CT/MRI medical data)

A cool fact: WebGL isn’t just for graphics—it’s often used for general-purpose GPU computing (GPGPU). In scenarios like deep learning inference or physics simulation, WebGL shaders can be treated as “CUDA in the browser.” The next-gen WebGPU goes further, natively supporting Compute Shaders, enabling AI inference directly in the browser.

Three.js: WebGL’s “Human-Friendly Wrapper”

Writing raw WebGL is excruciatingly verbose—rendering a rotating cube takes hundreds of lines. What Three.js does is wrap concepts like scene, camera, lights, materials, and geometry into abstractions, turning 3D into “declare a scene, add a few meshes, set up a camera”:

javascript

const scene = new THREE.Scene();
const camera = new THREE.PerspectiveCamera(75, width/height, 0.1, 1000);
const renderer = new THREE.WebGLRenderer();
const cube = new THREE.Mesh(new THREE.BoxGeometry(), new THREE.MeshBasicMaterial({ color: 0x00ff00 }));
scene.add(cube);
// Render loop: renderer.render(scene, camera) inside requestAnimationFrame

Does a blog need Three.js? Almost never. Unless your article topic is 3D data and readers need to rotate, zoom, and click on 3D objects in the browser. The cost of introducing Three.js is non-trivial (size, rendering performance, mobile compatibility). Don’t use it for “looking cool” in a blog.

WebGPU: Not an Upgrade to WebGL

WebGPU is an entirely new proposal. Its goal isn’t “WebGL 3.0”—it’s “Vulkan in the browser.” Key differences:

  • Compute Shaders: can run general-purpose GPU computation (AI inference, physics simulation); WebGL is graphics-only
  • Multi-threaded command submission: multiple CPU threads can submit commands to the GPU in parallel
  • Explicit resource management: the developer manages GPU memory, reducing implicit driver overhead

Performance benchmark: Babylon.js same-scene comparison shows WebGPU gains ~50 FPS, with CPU time dropping from 63ms to 0.2ms.

But WebGPU is currently only a W3C Working Draft, with incomplete browser coverage (Chrome 113+, Firefox 141+, Safari 26+). And it’s not a replacement for WebGL—they’ll coexist long-term. Engineering practice: build a WebGPU → WebGL → Canvas progressive enhancement fallback chain.

At this tier: you’ve reached the upper boundary of frontend graphics capability. For 99% of blog and web application scenarios, the WebGL/WebGPU tier isn’t needed—it’s reserved for data-intensive visualization, 3D applications, or cutting-edge GPU computing.

Animation Tier: From Static to Dynamic, Step by Step

Everything so far has been about “what to draw.” This section is about “how to make it move.” Animation isn’t decoration—in data visualization, it serves the dual purpose of perceived performance optimization and data narrative.

CSS Animations: The Zero-Dependency Starting Point

For the vast majority of blog animation needs, CSS animations are enough. No third-party libraries required. Want a pulsing SVG icon? Here:

css

.pulse {
  animation: pulse 2s ease-in-out infinite;
}
@keyframes pulse {
  0%, 100% { transform: scale(1); opacity: 1; }
  50% { transform: scale(1.2); opacity: 0.7; }
}
@media (prefers-reduced-motion: reduce) {
  .pulse { animation: none; }
}

CSS animation advantages: browser-native optimization, runs on the compositor thread independent of the JS main thread, supports prefers-reduced-motion accessibility preference.

Real case: TCP congestion window evolution diagrams. In this blog’s TCP series, I used CSS-driven SVG animations to show how congestion windows for Reno, CUBIC, BBR, and other algorithms evolve over time. Each algorithm gets its own <svg>, using stroke-dasharray + stroke-dashoffset for a “progressively draw the line” effect, with CSS @keyframes controlling the animation rhythm. When readers scroll to the chart, the congestion window curve “grows” before their eyes—visually conveying the core concept that “network state is dynamic.”

Limit: CSS animations excel at simple state transitions and looping animations. If you need precise multi-element timing orchestration (draw A first, wait 0.3s then draw B, B finishing triggers C’s morphing), CSS animations become painful—you’re stuck hard-coding animation-delay values with no timeline concept.

GSAP: Professional-Grade Timing Orchestration

GSAP’s core abstraction is the Timeline—chain multiple animations in series or parallel into one timeline, with precise control over every step:

javascript
1
2
3
4

const tl = gsap.timeline();
tl.to(".line-a", { strokeDashoffset: 0, duration: 1 })
  .to(".line-b", { strokeDashoffset: 0, duration: 0.8 }, "-=0.3") // start 0.3s early
  .to(".label", { opacity: 1, duration: 0.5 });

GSAP can also tween SVG’s d attribute (path shape morphing)—something CSS can’t do. Its ScrollTrigger plugin enables scroll-driven animations—the further the reader scrolls, the more frames the animation advances.

When to introduce GSAP? When you find yourself writing more than 5 animation-delay values in CSS to orchestrate multiple elements. Or when you need SVG shape morphing or scroll-driven animation. The cost of GSAP is ~30KB (core + ScrollTrigger)—well worth it for what it gives you.

Lottie: The Designer-to-Developer Animation Bridge

Lottie’s positioning is different from everything above—it’s not for developers to create animations, but to play animations that designers made in After Effects, in the browser.

Workflow: designer creates animation in AE → Bodymovin plugin exports JSON → frontend drops a <div>, calls lottie.loadAnimation(). Designers don’t learn code, developers don’t learn AE.

Use cases: loading animations, empty state illustrations, brand micro-interactions. Not suitable for data-driven geometric mapping—AE animations can’t react to changing data; for that, use D3/GSAP/Canvas.

Blog scenario: LottieFiles platform has thousands of free animations. Download a JSON file, embed it in the page. Perfect for article illustration animations, loading states, and making 404 pages feel alive.

Animation selection rule: simple state transitions → CSS, multi-element orchestration → GSAP, designer deliverables → Lottie, data-driven morphing → D3/Canvas.

One last time, the most important takeaway: don’t choose technology by “modernity”—choose by “is it enough.” SVG isn’t worse than Canvas; Canvas isn’t weaker than WebGL. Each has its own data volume sweet spot and interaction model. Mermaid isn’t “basic”—it’s the optimal implementation of the diagram-as-code paradigm. For a blogger, it is the correct answer.

If you remember only one thing: whatever diagram your blog needs, use the simplest tool that exactly solves that need. One bit more is overhead; one bit less is insufficient.
